## 游戏规则说明

在一周内，`Day 1` 到 `Day 7` 每天都会产生一个单日得分。将这 7 天的得分相加后，即为该玩家本周的总得分。两名玩家之间的胜负，按照各自一周的总得分进行比较。

## 每日算分规则

每日算分时，首先检查首发 5 人：

- 如果这 5 名首发球员当天都有比赛且均未受伤，则这 5 人的得分总和直接作为当日得分。
- 如果首发中有人当天没有比赛，或处于受伤状态，则按照替补规则进行递补。

替补规则如下：

1. 当日最终有效阵容必须满足 `2BC + 3FC` 或 `3BC + 2FC`。如果无法满足这两种阵容之一，则不能按 5 人满额完成替补算分。例如，`4BC + 1FC` 不符合规则。
2. 替补必须严格按照替补席顺序依次递补，不能跳过顺位选择球员。
3. 如果递补后仍不足 5 人，则按照最终符合规则并成功进入有效阵容的球员得分，计算当日总分。

## 开发规范

1. 每次新增规则、修正规则或调整实现时，必须同步更新 `README.md`，保证文档与代码一致。
2. 每次修改代码前，必须先完整浏览相关链路，避免局部 patch 破坏底层逻辑。
3. 如果用户指出规则或实现有误，应以最新修正规则为准，并同步更新文档与实现。
4. 优先保证数据正确性，其次再优化展示效果。
5. 避免继续做零散补丁式修复，优先做结构性修正。

## 数据链路排查规范

球员或经理 UID 的处理必须遵循以下要求：

1. 所有 UID 在进入主逻辑前必须统一标准化，建议统一按字符串处理。
2. 所有数据源必须先转成 `uid => data` 的映射，再进行合并。
3. 所有 merge 必须按 UID 精确合并，不能按数组下标或位置合并。
4. 所有输出必须按原始 UID 顺序生成，不能依赖 `Object.keys()` 恢复顺序。
5. 严禁把 UID 当作数组索引使用。
6. 严禁依赖对象 key 顺序恢复结果顺序。
7. 严禁使用异步 `forEach + push` 方式汇总请求结果。
8. 每位球员或经理输出时都必须是新的对象，不能复用引用或沿用上一个 fallback 对象。

## 2026-03-26 Live Logic Update

   - 当前周总分展示不再直接沿用官网当前 event 的实时周分，而是先剥离官网当前 event 那一行，再换成本地按有效阵容规则重算后的 `today` 分数。

   - 没开则显示 `None`

   - 以当前周总分为基础
   - 叠加未来剩余比赛的预期得分

   - 未来单场预期会优先参考 `form / 10`，并与 `points_per_game / 10`、`ep_next / 10` 做平滑混合，避免概率过度极端
   - 未来每天仍按有效 5 人阵容规则估算
   - 若本周队长卡未使用，当前按固定 `+65` 资源价值计入预估
   - 若本周仍有剩余 `FT`，当前按每个 `FT = +50` 的资源价值计入预估
   - 概率曲线会刻意放缓，并限制在较保守的区间内，避免轻易出现 `1% / 99%`
   - 该胜率为娱乐化预测，不代表真实胜负结果

   - 持有率只显示百分比
