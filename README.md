# nba-fantasy-h2h

## 游戏规则说明

在一周内，`Day 1` 到 `Day 7` 每天都会产生一个单日得分。将这 7 天的得分相加后，即为该玩家本周的总得分。两名玩家之间的胜负，按照各自一周的总得分进行比较。

## 每日算分规则

每日算分时，首先检查首发 5 人：

- 如果这 5 名首发球员当天都有比赛且均未受伤，则这 5 人的得分总和直接作为当日得分。
- 如果首发中有人当天没有比赛，或处于受伤状态，则按照替补规则进行递补。

替补规则如下：

1. 当日最终有效阵容必须满足 `2BC + 3FC` 或 `3BC + 2FC`。如果无法满足这两种阵容之一，则不能按 5 人满额完成替补算分。例如，`4BC + 1FC` 不符合规则。
2. 替补必须严格按照替补席顺序依次递补，不能跳过顺位选择球员。
3. 如果递补后仍不足 5 人，则按照最终符合规则并成功进入有效阵容的球员得分，计算当日总分。

## 开发规范

1. 每次新增规则、修正规则或调整实现时，必须同步更新 `README.md`，保证文档与代码一致。
2. 每次修改代码前，必须先完整浏览相关链路，避免局部 patch 破坏底层逻辑。
3. 如果用户指出规则或实现有误，应以最新修正规则为准，并同步更新文档与实现。
4. 优先保证数据正确性，其次再优化展示效果。
5. 避免继续做零散补丁式修复，优先做结构性修正。

## 数据链路排查规范

球员或经理 UID 的处理必须遵循以下要求：

1. 所有 UID 在进入主逻辑前必须统一标准化，建议统一按字符串处理。
2. 所有数据源必须先转成 `uid => data` 的映射，再进行合并。
3. 所有 merge 必须按 UID 精确合并，不能按数组下标或位置合并。
4. 所有输出必须按原始 UID 顺序生成，不能依赖 `Object.keys()` 恢复顺序。
5. 严禁把 UID 当作数组索引使用。
6. 严禁依赖对象 key 顺序恢复结果顺序。
7. 严禁使用异步 `forEach + push` 方式汇总请求结果。
8. 每位球员或经理输出时都必须是新的对象，不能复用引用或沿用上一个 fallback 对象。

## 当前已知维护要求

1. 玩家数据错误排查需覆盖：UID 遍历、异步请求汇总、数据映射、数组/对象使用方式，以及最终返回前端前的数据组装链路。
2. 转会趋势头像如果没有稳定、明确的数据源，则应直接移除，不保留半成品展示。
   当前实现策略：已移除前端头像半成品展示，后续如需恢复，必须先确认稳定图片源与 UID/球员 ID 映射方式。
3. FDR 颜色规则必须满足：
   - 红色表示难赛程
   - 绿色表示易赛程
4. FDR 需要使用未来 3 周的滚动窗口，而不是固定周数。
   当前实现策略：前端必须通过 API 动态刷新 FDR，不能只在首屏静态加载一次。
5. FDR 的最终目标评分规则为：
   - 联赛总排名权重 `70%`
   - H2H 联赛排名权重 `30%`
## 2026-03-25 维护更新

1. 当前线上后端已集成到 `worker/src/index.js`，`backend/` 仅保留为历史本地版本，不作为线上主逻辑维护。
2. 为避免中文队名或别名导致的错配，H2H 与 FDR 的赛程数据改为按 `UID` 维护，不再依赖队名字符串反查 UID。
3. FDR 的联赛排名来源改为 `https://nbafantasy.nba.com/api/entry/{entry_id}/` 中联赛 `1653` 对应 classic league 的 `entry_rank`。
4. FDR 颜色规则明确为：
   - `5` = 最难，深红色
   - `4` = 偏难，红色
   - `3` = 中性，灰色
   - `2` = 偏易，浅绿色
   - `1` = 最易，深绿色
5. 今日赛程状态现在优先结合 `finished` / `finished_provisional` / `started`，并对长时间未更新的比赛做结束兜底，减少“比赛已结束但仍显示进行中”的情况。
6. FDR 模块下新增 league 统计：
   - 今日平均人数：当天有比赛的 roster 人数平均值，不区分是否可算分
   - 今日平均有效人数：当天有比赛且未伤停、理论上可算分的人数平均值
7. 前端默认改为同域 API 调用：`window.__API_BASE__ = ""`，优先走 `pages.dev` 域名下的 `/api/*`，不再依赖直接访问 `workers.dev`。
8. 新增头部 Logo，资源文件为 `frontend/LOGO.jpg`。
9. 为兼容 Cloudflare 免费版 Worker 的外部子请求限制，后端刷新改为分块模式：
   - 默认 `POST /api/refresh` 只刷新一部分 UID
   - Worker 定时任务应高频执行，逐步完成整轮刷新
   - 如需全量刷新，可显式调用 `POST /api/refresh?mode=full`，但在免费版下可能触发子请求上限
10. `pages.dev` 同域 API 需要通过 `functions/api/[[path]].js` 代理到 `worker/src/index.js`。如果 Pages 项目未绑定 `NBA_CACHE` 和 `REFRESH_TOKEN`，前端会拿到 HTML 页面或 500，而不是 JSON。
